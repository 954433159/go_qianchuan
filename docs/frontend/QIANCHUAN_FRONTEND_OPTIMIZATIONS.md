# 前端分阶段优化方案（含验收标准）

整体策略: 以“域”为单位分期推进。每期目标是“从入口到提交”的最小可用闭环，并保持 UI/交互的一致性与错误可观察性。

Phase 1（1-2 周）— 报表与路由体验兜底
- 路由兜底: 增加 403 与 404 专用页面（含跳转 CTA）并在 `App.tsx` 配置
- 报表基础: 为 Reports 增加“基础版/高级版”切换位（保留禁用态），并为未实现报表添加占位组件（含 docs 链接）
- Sentry: 在 Reports 与 Finance 流程关键节点添加面包屑（页面进出、导出、请求失败）
验收: 新增 403/404 页；Reports 页展示清晰；Sentry 事件含业务面包屑

Phase 2（2-3 周）— 全域推广最小闭环
- 接口/页面: 列表/创建/详情/编辑/状态切换链路打通（按后端可用范围渐进）
- 表单: 分步表单 + 审核校验 + 保存草稿（本地态）
- 组件: 复用 Ad/Campaign 表单段落组件，减少重复
验收: 能创建/查看/编辑/更新状态；错误与校验提示统一；回退/保存可用

Phase 3（2-3 周）— 报表扩展/素材分析
- 实现搜索词、素材报表、视频流失用户等 2-3 个典型报表（按静态页）
- 通用报表框架: 过滤器、列配置、导出、空态、分页、骨架、错误态
- 大表虚拟化与服务端分页
验收: 至少 2 个新报表上线；报表框架复用到旧报表

工程化与质量（贯穿各期）
- 测试: 引入 MSW 建立 API Mock；为 3 个页面补充集成测试（含路由）
- 性能: 切分 chunk；路由级 prefetch；关键页面使用 React.memo/useCallback 优化
- 文档: 为组件与页面补充注释与 README；在 docs/frontend 下沉淀实践
